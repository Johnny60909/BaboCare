## Why

現有系統缺乏寶寶資料管理功能。不同角色（系統管理員、保母、家長）需要根據他們的責任範圍管理和查看寶寶資訊。此變更引入寶寶資料的完整生命週期管理，包括後台的CRUD操作和前端的角色感知寶寶顯示，同時在系統核心實現統一的審計跟蹤機制。

## What Changes

- **後台寶寶管理介面**：系統管理員和保母可以新增、編輯、查看所有寶寶資料
- **寶寶資料編輯**：包含寶寶名稱、出生日期、性別，以及家長和保母帳號的關聯
- **角色感知的寶寶顯示**：首頁顯示根據登入帳號類型不同的寶寶故事列
  - 系統管理員可以看到全部寶寶
  - 保母只能看到他負責照顧的寶寶，或建立新的寶寶
  - 家長只能看到自己的寶寶
- **寶寶與人員的關聯**：建立寶寶與多位家長和單一保母的關係
- **審計欄位**：所有主要聚合根實體都包含統一的審計欄位（CreatedAt、CreatedBy、UpdatedAt、UpdatedBy），透過介面進行管理
- **智能預帶**：如果系統中只有1位保母或1位管理員，編輯寶寶時自動預帶為照護者

## Capabilities

### New Capabilities

- `baby-profile-management`: 寶寶資料的建立、編輯、查看功能，包含名稱、出生日期、性別、與家長和保母的關聯
- `baby-access-control`: 基於用戶角色的寶寶訪問控制，確保只有授權的使用者能查看特定的寶寶資訊

### Modified Capabilities

- `admin-account-management`: 新增寶寶管理功能到後台，允許在帳號管理介面中管理寶寶資訊

## Impact

- **後端變更**：
  - 新增 Baby 實體及其聚合根
  - 新增 BabyNanny、BabyParent 關聯實體
  - 實現 IAuditable 介面用於審計欄位管理
  - 新增 Baby 管理的 API 端點 (GET、POST、PUT、DELETE)
  - 實現寶寶訪問控制的授權策略

- **前端變更**：
  - 後台新增寶寶管理頁面
  - 首頁修改以支援寶寶故事列的動態顯示
  - 寶寶編輯表單（名稱、出生日期、性別、家長選擇、保母選擇）

- **資料庫變更**：
  - 新增 Babies 表
  - 新增 BabyNannies 和 BabyParents 關聯表
  - 所有主要實體表添加審計欄位

- **受影響的角色**：系統管理員、保母、家長

## Rollback Plan

如需回滾此變更：

1. 刪除所有寶寶管理相關的 API 端點和前端頁面
2. 從資料庫移除 Babies、BabyNannies、BabyParents 表（或標記為過時）
3. 從現有實體中移除 IAuditable 實現（如果需要完全回滾）
4. 恢復前端首頁至原始的非寶寶顯示邏輯

## Affected Teams

- **後端團隊**：實現 Baby 實體、API 端點、訪問控制邏輯
- **前端團隊**：實現後台寶寶管理頁面和首頁寶寶顯示
- **資料庫/DevOps 團隊**：執行數據庫遷移和部署
