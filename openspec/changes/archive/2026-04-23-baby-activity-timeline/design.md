## 背景

BaboCare 允許保母和系統管理員追蹤和分享寶寶活動。目前首頁僅顯示基本寶寶資訊，缺乏日常活動時間軸。本設計確立如何實現一套活動創建、存儲、檢索和展示系統，維持時間順序並提供轉盤介面瀏覽個別寶寶活動。

現有可利用的功能：

- 用戶身份驗證和基於角色的存取控制（identity 模組）
- 寶寶檔案管理
- 媒體處理（圖片裁切、存儲）

## 目標 / 非目標

**目標：**

- 讓保母快速記錄寶寶活動，附帶類型特定選項
- 在首頁動態牆顯示活動，按最新→舊排序
- 提供轉盤導航瀏覽單一寶寶的活動歷史
- 根據最新活動時間戳重新排序寶寶頭像
- 所有活動類型均支援媒體（照片）和文字備註

**非目標：**

- 活動編輯或刪除（v1 不含）
- 實時通知或活動評論
- 活動搜尋或篩選
- 逐個活動的隱私控制（使用現有授權模式）
- 分析或趨勢可視化

## 技術決策

### 1. 活動類型枚舉 vs. 多態表設計

**決策**：使用區分聯合模式，搭配 `Type` 枚舉和靈活的 JSON 選項欄位。

**理由**：

- 餵奶、進食、尿布、睡眠、心情具有重疊的結構（照片、備註）且具有類型特定選項
- 將各類型存儲在分離表中會增加查詢複雜度
- 單一 `Activity` 表搭配 `Type` 枚舉和 `Options` JSON 對初版實現更簡潔
- 選項形狀由 API 層根據 Type 進行驗證

**考慮的替代方案**：

- 各類型分離表（增加 JOIN 複雜度）
- 完整多態模式（Entity Framework Table Per Type - 增加遷移複雜度）

### 2. 活動動態牆排序與頭像重新排序

**決策**：按 `CreatedAt DESC` 排序動態牆；頭像順序基於各寶寶最新活動的 `CreatedAt`。

**理由**：

- 單一排序鍵簡化 API 和前端
- 最新活動時間戳自然排序頭像
- 無需額外的排名或置頂邏輯

**考慮的替代方案**：

- 按 `ModifiedAt` 排序（但 v1 活動不可變）
- 基於活動新近度的加權分數（v1 過度設計）

### 3. 活動轉盤導航

**決策**：前端管理轉盤狀態；API 提供 `GET /babies/{babyId}/activities?limit=20` 按日期 DESC 排序。

**理由**：

- 降低後端複雜度
- 前端可本地快取和導航
- 單一 API 呼叫獲取寶寶最近活動
- 左/右導航在 React 狀態中更新轉盤索引

**考慮的替代方案**：

- 使用 prev/next 游標進行分頁（對典型活動量不必要）

### 4. 媒體存儲

**決策**：重用現有圖片裁切/上傳流程；將活動照片路徑存儲在 `PhotoUrl` 欄位中。

**理由**：

- 與現有寶寶檔案照片處理一致
- 已在前端和後端實現
- 所有活動都需要照片；適用相同驗證

### 5. 權限模型

**決策**：活動繼承寶寶的權限範圍。用戶可為其具有保母或管理員存取權的寶寶創建/查看活動。

**理由**：

- 符合現有 `baby-access-control` 規範
- 不需要新的權限概念
- 使用現有授權上下文

### 6. 活動創建進入點

**決策**：首頁底部導覽列的「+」按鈕開啟活動創建頁面；頁面以頭像按鈕排列顯示寶寶（非下拉清單），點選後以邊框高亮標示已選寶寶。

**理由**：

- 行動優先設計中可見且直觀
- 頭像按鈕提供更直覺的視覺回饋，符合 IG 風格
- 符合現有導覽列模式

### 7. 照片顯示策略

**決策**：動態牆卡片與上傳預覽均採用動態高度顯示，不強制 1:1 裁切，最大高度限制 600px。

**理由**：

- 強制 aspect-square 會裁切直式照片，遺失重要內容
- `max-w-full h-auto max-h-[600px]` 保持原始比例同時防止過高的卡片破壞動態牆節奏
- 超過 600px 時瀏覽器等比縮放，無需 JavaScript 計算

### 8. 轉盤 Modal 層級與 UX

**決策**：CarouselModal 使用 `z-[200]` 確保蓋住底部導覽列（`z-50`）；箭頭定位於 `fixed` 容器而非圖片容器；左上角提供 X 關閉鈕。

**理由**：

- `z-50` 無法蓋住同層級的底部導覽列，需明確提升層級
- 箭頭放在 `flex` 子容器的 `relative` 內會因 flex 計算產生跑版，改定位於最外層 `fixed` 容器可靠
- X 關閉鈕比僅支援點擊背景關閉更直觀

## 風險 / 權衡

| 風險                                                            | 緩解方案                                         |
| --------------------------------------------------------------- | ------------------------------------------------ |
| **照片上傳失敗但活動已創建** → API 呼叫分離（呼叫可能部分成功） | 在活動提交前要求上傳照片；將缺少照片視為驗證錯誤 |
| **高活動量導致動態牆查詢變慢**                                  | 在 `CreatedAt` 上建立索引 + 分頁限制             |
| **並發活動創建**                                                | 資料庫處理唯一約束；前端顯示樂觀更新             |
| **選項 JSON 驗證**                                              | 為各活動類型實現類型化 DTO；在控制器層驗證       |
| **頭像重新排序導致卡頓**                                        | 限制頭像查詢；React 透過鍵高效地重新渲染         |

## 部署計畫

1. **階段 1**：部署後端 Activity 實體、服務和 API 端點（創建、取得動態牆、取得寶寶活動）
2. **階段 2**：部署資料庫遷移（Activity 表 Schema）
3. **階段 3**：部署前端活動創建頁面和轉盤組件
4. **階段 4**：更新首頁以顯示動態牆和重新排序的頭像
5. **階段 5**：將「+」按鈕新增到底部導覽列

**回退策略**：

- 功能旗標 `ENABLE_ACTIVITY_CREATION` 以禁用創建端點
- 透過旗標在首頁隱藏動態牆組件
- 將頭像排序恢復為原始排序

## 已決事項

1. **活動分頁限制**：採用動態加載模式 - 初始載入 20 筆活動，使用者向下滑動時自動載入下一批
   - 前端在使用者滑動到動態牆底部時觸發 `GET /api/activities?skip=20&limit=20` 查詢
   - 無限滾動提供無縫瀏覽體驗

2. **視頻支援**：v1 僅支援照片
   - 所有活動類型只能上傳單張照片
   - 視頻支援延後至後續版本

3. **時區處理**：UTC 存儲、本地時區顯示
   - 後端以 UTC 存儲所有活動的 `CreatedAt` 時間戳
   - 前端從用戶的瀏覽器時區設定（`Intl.DateTimeFormat` 或用戶帳戶設定）取得本地時區
   - 顯示時將 UTC 轉換為本地時區
   - **時區設定**：使用者帳戶應有時區偏好設定，可在設定頁面更改；如未設定則使用瀏覽器時區

4. **重複活動防止**：前端按鈕狀態控制
   - 發布後立即禁用發布按鈕並顯示載入狀態（轉圈圈）
   - 防止使用者重複點擊
   - 後端可選使用幂等性鍵作為額外保護層，但 v1 主要依賴前端防護
